# Code Review — PitStrategy

**Audit date:** 2026-07-14
**Files reviewed:** 32 across Python, HTML, CSS, JavaScript, PyInstaller spec, Markdown (excludes `build/`, `dist/`, `__pycache__/`, `.pytest_cache/` — generated/vendored output, not reviewed line-by-line)
**Tools used:** `python -m py_compile` (clean), `ruff check` (clean, default rule set — no project ruff config exists), `ruff format --check` (informational only, see Low Issues), `python -m pytest` (70/70 passing), manual review. `bandit` and `mypy` are not installed in this environment and were skipped.

## Summary

This is a well-built, well-documented solo project — the README alone is some of the most thorough "why," not just "what," documentation I've seen in one of these audits, and it made this review much easier. Test coverage of the pure-computation core (`fuel.py`, `tires.py`, `relative.py`, `leader_pace.py`, `session_baseline.py`, `pit_planner.py`) is genuinely good: 70 tests that exercise real edge cases (refuel outliers, rolling-window aging, sentinel values), not smoke tests. No secrets, no injection vectors, no crash-class bugs found. The main things worth attention are one real cross-session state bug in the relative/rival tracker, an unverified telemetry-sentinel assumption inherited from a field that isn't rendered anywhere yet, and a couple of housekeeping items (no `.gitignore`, a stale packaging comment) that are cheap to fix now before they cause confusion later.

## 🟡 Medium Issues

- ~~**[Correctness] `RelativeTracker.has_pitted` survives session transitions**~~ **FIXED 2026-07-14** — added `RelativeTracker.reset()` (`core/relative.py`) and called it alongside the fuel/tire resets in `server/engine.py`'s session-transition block. Covered by a new `test_reset_clears_sticky_pit_status` in `tests/test_relative.py`.
  `StrategyEngine._tick()` resets `self._fuel`, `self._tires`, and the lap-time trackers on a session transition (`self._fuel.reset()`, `self._tires.reset()`, etc. at engine.py:138-142), but never touches `self._relative`. `RelativeTracker._has_pitted` is a dict keyed by `car_idx` that's set once and never cleared (`update_pit_status()`, relative.py:23-26 — deliberately sticky within a session, per its own docstring and `test_has_pitted_is_sticky`). Across a real session change (e.g. qualifying → race, or race → race in a multi-race event), that dict is never reset, so a car that visited pit road during qualifying will show a stale `PIT` tag at the start of the race even though it hasn't pitted this stint. Fix: add a `self._relative = RelativeTracker()` (or a `reset()` method on the class) alongside the other resets in the session-transition block.

- **[Correctness, unverified] `SessionLapsRemainEx`'s "unlimited" sentinel isn't special-cased** — `core/irsdk_client.py:117`, `server/engine.py:152`
  `state.laps_remain` is read straight from `self._ir["SessionLapsRemainEx"]` and fed directly into `laps_remaining_in_race` with no bounds check. iRacing's own SDK convention for this field is to return a large sentinel (commonly `32767`) for practice/qualifying/unlimited-time sessions rather than `None` — flagging this as an assumption since, like the relative-panel/tire-temp channels the README already calls out as unverified against a live session, I haven't confirmed this against real telemetry either. If that's the actual behavior, `laps_remaining_in_race`, `total_fuel_to_finish`, and `compute_live_strategy()`'s `strategy` field would all silently produce nonsense (e.g. "fuel needed to finish: 78,000 L") during any non-race session. Impact today is low since none of those three fields are currently rendered by `overlay.js` (only `laps_remaining_leader_pace`, which is derived from `session_time_remain` instead, is used) — but they're already serialized over `/ws/live` every tick, and `strategy` is explicitly kept "for something else to use later" per the README. Worth a quick check against a real qualifying session, and a sentinel guard (e.g. treat anything above a sane lap-count ceiling as `None`) if confirmed.

- **[Correctness, unverified] `SessionLapsRemainEx`'s "unlimited" sentinel isn't special-cased** — NOT FIXED, left as-is

Revisit once this can be checked against live telemetry.

- ~~**[Correctness/Diagnostics] Caught exceptions are silently discarded instead of surfaced to the console**~~ **FIXED 2026-07-14** — both handlers now `print(..., file=sys.stderr)` + `traceback.print_exc()` before falling back to the disconnected state.
  `StrategyEngine._run()` catches any exception from `_tick()` and stores `{"connected": False, "error": str(exc)}` in `self._latest` — but `overlay.js`'s `render()` never reads `data.error` (grepped: zero references), so that message goes nowhere. `IRSDKTelemetrySource._run()`'s own catch-all is worse: it just sets `connected = False` with no message captured at all. The README explicitly says the packaged exe keeps `console=True` specifically "since startup errors... are much easier to diagnose with visible output than silently failing behind a windowed app" — but neither of these handlers actually prints/logs anything, so a real bug (e.g. exactly the kind of wrong-channel-name mismatch the README itself flags as an open risk for the relative panel or tire-temp channels) would show up only as an unexplained "disconnected" state with zero diagnostic trail, defeating the stated purpose of the visible console. Cheap fix: `print(f"[PitStrategy] tick error: {exc}", file=sys.stderr)` (or `traceback.print_exc()`) in both handlers.

- ~~**[Packaging] `pitstrategy.spec`'s own comment describes machinery that doesn't exist**~~ **FIXED 2026-07-14** — comment rewritten to match `run.py`'s actual (correct) description of the `--webview-window`/`sys.frozen` dispatch mechanism.
  The spec's header comment says a frozen exe is re-invoked with `-m server.webview_launcher <role> <url>`-equivalent args and that "`server/__main__.py` exists... a frozen exe has no real `python -m`... so `PitStrategy.exe -m server.webview_launcher overlay <url>` is handled by `__main__.py`'s shim." There is no `server/__main__.py` in the project (confirmed via search), and the actual mechanism (correctly documented in `run.py`'s own module docstring and the README) is different: `server/browser.py` invokes `sys.executable --webview-window <role> <url>`, which `run.py`'s `main()` intercepts directly before argument parsing. The spec comment appears to describe an earlier approach that was superseded but never updated here. Not a functional bug (the actual code works and is documented correctly elsewhere), but it's exactly the kind of stale comment that will send future-you down the wrong path when debugging a packaging issue — worth syncing to match `run.py`'s docstring.

- ~~**[Structure/Hygiene] No `.gitignore`, and `build/`/`dist/` (67 MB of PyInstaller output) sit in the project root alongside source**~~ **FIXED 2026-07-14** — the project became a real git repo the same day this was flagged; `.gitignore` (covering `build/`, `dist/`, `__pycache__/`, `.pytest_cache/`, `.ruff_cache/`, `.claude/`) was added before the first commit.
  This isn't a git repo yet, so nothing is at risk of being committed today, but there's also no `.gitignore` staged for when it becomes one. `build/` (28 MB) and `dist/` (39 MB) are exactly the kind of generated artifacts that get accidentally committed the first time a repo is initialized with a broad `git add .`. Worth adding a `.gitignore` now (covering `build/`, `dist/`, `__pycache__/`, `*.pyc`, `.pytest_cache/`, and `%APPDATA%`-style local settings if a future dev environment ever colocates them) before this project's git history exists, rather than after.

- ~~**[Testing] `server/settings_store.py` has zero test coverage**~~ **FIXED 2026-07-14** — added `tests/test_settings_store.py` (6 tests: defaults-when-missing, round-trip, parent-dir creation, corrupt-JSON fallback, unknown-field tolerance, missing-field defaults), isolated from the real `%APPDATA%` via a `monkeypatch`-based fixture.

## 🟢 Low Issues

- **[Security/Style] `overlay.js` builds relative-panel rows with unescaped `innerHTML`** — `server/static/overlay.js:181-198`
  `renderRelativeGroup()` interpolates `car.car_number` (sourced from `core/irsdk_client.py`'s `_read_car_numbers()`, i.e. other drivers' `DriverInfo.Drivers[].CarNumber` — data the player doesn't control) directly into an `innerHTML` template string. Practical risk is very low here: this is a single-user local desktop overlay, the `js_api` surface exposed to the page is a single harmless `resize()` method, and iRacing car numbers are conventionally short and constrained — but it's still interpolating other-session-participants'-controlled string data into `innerHTML` rather than `textContent`/`createElement`, which is the kind of thing worth fixing on the next pass through this file for free, not urgent enough to stop and fix now.

- **[Packaging] No dev/runtime dependency split for `pytest`** — `requirements.txt:6`
  `pytest` lives in the main `requirements.txt` alongside runtime deps, even though the project already has a `requirements-build.txt` convention for build-only tooling (`pyinstaller`). Moving `pytest` into a `requirements-dev.txt` (or similar) would make `requirements.txt` an accurate list of what the shipped exe actually needs at runtime.

- **[Packaging] No dependency lockfile; only open-ended ranges** — `requirements.txt`
  `fastapi>=0.110,<1`, `uvicorn[standard]>=0.29,<1`, etc. are wide ranges with no pinned/frozen snapshot. Reasonable for early solo development, but now that this is also being packaged into a distributable `.exe`, a `pip freeze`-style lock (or at minimum recording the exact versions used for a given build in the release notes) would make a specific build reproducible if a regression needs bisecting later. Currently-installed versions (fastapi 0.139.0, uvicorn 0.51.0, pydantic 2.13.4, pywebview 6.2.1, pyirsdk 1.3.6) all look current and none raised any known-CVE flags in a spot check.

- **[Structure] No linter/formatter config committed** — project root
  `ruff check` passes clean, but only because no `pyproject.toml`/`ruff.toml` exists, so it's running ruff's narrow default rule subset (E4/E7/E9/F) rather than anything opinionated (import order, complexity, etc.). `ruff format --check` reports all 17 non-trivial `.py` files would be reformatted under ruff's default style — that's not meaningful on its own (the current hand-formatting, with deliberate multi-line conditionals and long explanatory inline comments, is a legitimate style choice, not sloppiness), but it does mean there's currently no machine-enforced style at all; consider committing a `pyproject.toml` with an explicit `[tool.ruff]` config (even one that matches the current style, e.g. a wider `line-length`) so consistency is enforced rather than incidental.

- **[Structure] `/api/fuel-limit` and `/api/fuel-limit/clear` have no UI to invoke them** — `server/app.py:81-89`, `server/static/settings.html`
  `FuelTracker`'s manual fuel-limit override machinery is fully implemented and well-tested (`core/fuel.py`, `tests/test_fuel.py`), and the two endpoints that drive it exist and work, but nothing in `overlay.html`/`settings.html` calls them — they're currently only reachable via a raw HTTP request. This may simply be the same kind of "not wired up yet" state as `/api/open-overlay` (which the README explicitly documents as having "no page with a button wired to that endpoint... hit it directly" for now), and lines up with the settings-page work already noted as deferred. Worth either a similar one-line README callout, or folding it into the deferred settings-page work when that comes back around.

- **[Structure] `StrategyEngine._tick()` is a large, multi-purpose method** — `server/engine.py:107-232`
  At ~125 lines, `_tick()` now handles session-transition detection, fuel/tire updates, live-strategy computation, relative-panel updates, and leader-pace/fuel-margin math all in one method. It's still readable (good inline comments throughout), but as more per-tick features get added (this session added leader-pace and qualifying-baseline logic on top of what was presumably a smaller original), consider splitting the leader-pace/fuel-margin block (engine.py:177-211) into a small private helper (e.g. `_leader_pace_fields(state)`) purely for readability and unit-testability — that block is pure arithmetic with no side effects and currently has no direct test coverage of its own (only indirectly, via nothing — there's no `test_engine.py` at all).

- **[Structure] `async def` endpoints/lifespan perform blocking file I/O** — `server/settings_store.py:36-49`, `server/app.py:75-79`, `server/app.py:32`
  `load_settings()`/`save_settings()` do synchronous `Path.read_text()`/`write_text()` calls, invoked from inside `async def api_post_settings()` and the `async def lifespan()` startup. For a tiny local JSON file this blocks the event loop for microseconds and is genuinely not worth fixing today, but it's the kind of pattern worth switching to `anyio.to_thread.run_sync()` if this ever grows into something that does non-trivial I/O on the same request path.

## Dimensions with no findings

- **Security (secrets/injection):** No hardcoded secrets, API keys, or tokens anywhere in tracked source. No `eval`/`exec`/`pickle` on untrusted input, no SQL (no database at all), no `subprocess` calls with `shell=True`, no unsafe deserialization. The server binds to `127.0.0.1` by default and this is a single-user local tool, so the lack of any auth on its HTTP/WebSocket endpoints is appropriate, not a gap — flagging only as an assumption: if `--host` is ever pointed at `0.0.0.0`/a LAN interface, that changes, but nothing in the current code does that by default.
- **Optimization & performance:** No quadratic loops over meaningfully-sized data, no repeated I/O/network calls in hot loops, no obvious pandas/numpy-style anti-patterns (none of those libraries are used). The 5 Hz compute loop / 0.5 s websocket push / 10 Hz iRacing poll are all sensible, deliberately-chosen rates per the code's own comments.
- **Syntax & correctness (general):** No mutable-default-argument bugs (dataclasses consistently use `field(default_factory=...)`), no off-by-ones spotted in the well-tested `core/` arithmetic, no unreachable code, no obvious resource leaks (the one background thread + subprocess pattern is consistently start/stop/join'd).
- **Concurrency:** The one piece of genuinely cross-thread mutable state (`FuelTracker`'s fuel-limit override, written from a request-handling thread and read from the engine's background poll thread) is correctly guarded by its own dedicated lock, with a comment explaining exactly why. `StrategyEngine._latest` is similarly lock-protected. No races found.
